Phase 2 — Embedding
2
Vectorisation des chunks Voyage AI

Chaque chunk de texte est envoye a l'API Voyage AI pour obtenir un vecteur dense de 1024 dimensions. Les chunks sont envoyes par batch de 128 pour optimiser le debit. En mode hors-ligne, fallback sur Ollama nomic-embed-text (768d).

Batch BuilderRegroupe les chunks par lots de 128
Chaque lot ≤ 120,000 tokens
→
Voyage APIPOST https://api.voyageai.com/v1/embeddings
Model: voyage-3-lite
Retour: float[1024] par chunk
↓
SQL — Stockage vecteurUPDATE chunks SET embedding = ? WHERE id = ?
// embedding = Buffer.from(Float32Array) — BLOB compact
// Voyage AI request { "model": "voyage-3-lite", "input": ["chunk_1 text...", "chunk_2 text...", ...], "input_type": "document" } // Response { "data": [{ "embedding": [0.012, -0.034, ...], "index": 0 }, ...], "usage": { "total_tokens": 8420 } }
⚠ Decision : Online / Offline

Au demarrage, le serveur teste la connectivite internet via un appel HEAD https://api.mistral.ai/v1/models (timeout 3s). Le resultat determine quel provider d'embedding est utilise :

Voyage AI (1024d)
→ offline →
Ollama nomic-embed-text (768d)

Les vecteurs de dimensions differentes ne sont pas mixables. Si un workspace a ete cree online, il ne peut pas etre interroge offline (et inversement). Le champ chunks.embed_model trace le modele utilise.

3
Schema SQLite mis a jour Backend

La table chunks recoit deux nouvelles colonnes. Les tables BM25 (chunk_terms) sont conservees comme fallback lexical.

4
Embedding de la question Voyage AI

La question de l'utilisateur est convertie en vecteur avec le meme modele que les chunks du workspace. Appel unique a l'API Voyage AI (ou Ollama en offline).

5
Recherche vectorielle (cosine similarity) Vector Search

Calcul de similarite cosinus entre le vecteur de la question et tous les vecteurs du workspace. Tri par score decroissant, top-K = 8 chunks. Brute-force en memoire — performant jusqu'a ~50,000 chunks.

SQL — ChargementSELECT id, document_id, content, embedding FROM chunks
WHERE workspace_id = ? AND embedding IS NOT NULL
↓
Cosine Similaritycos(q, c) = dot(q, c) / (||q|| * ||c||)
Calcul natif Float32 en Node.js
→
Tri & SelectionSort desc par score
Top 8 chunks retenus
// Fonction cosineSimilarity en JS function cosineSimilarity(a, b) { let dot = 0, normA = 0, normB = 0; for (let i = 0; i < a.length; i++) { dot += a[i] * b[i]; normA += a[i] * a[i]; normB += b[i] * b[i]; } return dot / (Math.sqrt(normA) * Math.sqrt(normB)); }

2. Comparaison Online / Offline

Sailor detecte automatiquement la connectivite au demarrage et toutes les 60 secondes. Aucune intervention manuelle requise. Le mode change dynamiquement.

Online — Mistral + Voyage AI
1
Embedding : Voyage AI voyage-3-lite — 1024 dimensions, optimise pour le francais
2
Recherche : Cosine similarity sur vecteurs 1024d
3
Generation : mistral-large-latest via Mistral API — 128k tokens context, multilingue natif
4
Qualite : Reponses riches, nuancees, sources precises. Ideal pour usage courant.
Offline — Ministral 3B + Nomic Embed
1
Embedding : Ollama nomic-embed-text — 768 dimensions, tourne 100% local
2
Recherche : Cosine similarity sur vecteurs 768d + fallback BM25 si pas de vecteurs
3
Generation : ministral:3b via Ollama — 32k tokens context, RAM ≤ 4GB
4
Qualite : Reponses correctes, formatage plus simple. Zero dependance reseau.

6. Plan d'implementation

Etape 1 — Migration schema & embeddingsBackend

Ajouter les colonnes embedding, embed_model, embed_dim a la table chunks. Creer les fonctions embedWithVoyage(), embedWithOllama(), cosineSimilarity(), searchVectors().

Etape 2 — Detection connectivite & routing IABackend

Fonction checkConnectivity() appelee au boot et toutes les 60s. Route vers Mistral API ou Ollama automatiquement. Le changement est transparent pour le frontend.

Etape 3 — Integration Mistral APIAI

Remplacer aiChatPremium() (Anthropic) par appel Mistral API. Chaine de fallback : mistral-large-latest → mistral-small-latest → ministral:3b local.

Etape 4 — Embedding async a l'uploadBackend

Apres l'upload, lancer l'embedding en arriere-plan (non bloquant). Le frontend affiche une progression. Le chat est disponible des que les premiers chunks sont embedes (fallback BM25 sinon).

Etape 5 — Tests & validationFrontend

Tester avec le dataset mutuelle (50 docs). Verifier la qualite des reponses en mode Mistral vs Ministral. Valider le fallback automatique en coupant le reseau.
